# radar-dev – работа с репозиторием Radar

Ты – инженер проекта Radar. Этот скилл описывает, как понимать структуру репозитория, запускать сервис, вносить изменения, обновлять README, проверять ошибки и готовить проект к публикации. Отвечай на русском, соблюдай типографику: тире – en-dash (–), без буквы е с точками.

## Что такое Radar

Конвейер из пяти шагов: сбор из ~85 Telegram-каналов → фильтрация шума → анализ двумя агентами → композиция дайджеста → публикация в Telegram-канал. UI нет, только данные и агенты. Подробное описание – в `PROJECT.md`, обзор – в `README.md`.

## Структура репозитория

```
sources/channels.json     # список источников: id, url, priority, tags, active
scripts/
  fetch-posts.py          # шаг 1: сбор из Telegram (Telethon/MTProto)
  filter-signals.py       # шаг 2: фильтрация шума через OpenAI (флаг --date для backfill)
  run-pipeline.py         # оркестратор: fetch → filter → digest, флаги --skip-*
  publish-post.py         # публикация поста из posts/ в Telegram-канал (Bot API)
  load-to-sqlite.py       # вспомогательная загрузка данных в SQLite
  import-tg-folders.py    # импорт списка каналов из экспорта папок Telegram
agents/
  digest-agent.py         # шаг 5-подготовка: сборка дайджеста
  post-adapter.md         # промпт скилла адаптации постов
data/raw/                 # сырые сообщения (только чтение, не менять после сохранения)
data/processed/           # отфильтрованные сигналы
data/analysis/            # разборы агентов (aitech / legal) по датам
digests/                  # готовые дайджесты YYYY-MM-DD-digest.md
posts/                    # адаптированные посты под площадки (JSON)
.claude/agents/           # analyst-legal, analyst-aitech, digest-composer
.claude/commands/         # run-analysis и др.
.claude/skills/           # post-adapter, radar-dev
```

Правила нейминга (см. `CLAUDE.md`): только строчные латинские буквы и дефис, дата в начале для дайджестов и сырых данных, никакой кириллицы и пробелов в именах файлов.

## Как запустить локально

```bash
python -m venv .venv
source .venv/bin/activate          # Windows: .venv\Scripts\activate
pip install -r requirements.txt
cp .env.example .env                # заполнить ключи
python scripts/run-pipeline.py
```

Частичные прогоны:

- `python scripts/run-pipeline.py --skip-fetch` – без сбора, только фильтр и дайджест.
- `python scripts/run-pipeline.py --skip-filter --skip-digest` – только сбор.
- `python scripts/filter-signals.py --date 2026-05-20` – backfill конкретной даты.
- `python agents/digest-agent.py --date 2026-05-20` – пересобрать дайджест за дату.

Результат прогона – файл `digests/YYYY-MM-DD-digest.md`.

## Секреты

Все ключи – только через переменные окружения или `.env` (в `.gitignore`, не коммитить):
`TELEGRAM_API_ID`, `TELEGRAM_API_HASH`, `TELEGRAM_PHONE`, `TELEGRAM_SESSION_STRING`, `TELEGRAM_BOT_TOKEN`, `TELEGRAM_CHAT_ID`, `OPENAI_API_KEY`. Перед коммитом убедись, что ключей нет в коде, логах и данных.

## Как вносить изменения

- **Добавить источник:** внести объект в `sources/channels.json` (id, name, url, type, priority, tags, active). Проверить релевантность и активность канала. Обновить поле `updated`.
- **Правка фильтрации/промптов:** логика фильтра – в `scripts/filter-signals.py`, промпты агентов – в `.claude/agents/`. Менять по одному шагу и проверять прогоном с `--skip-fetch`.
- **Сырые данные не изменять:** `data/raw/` только на чтение. Дедупликация – на этапе обработки в `data/processed/`.
- **Деструктивные операции** (удаление, перезапись) – только с явного подтверждения человека.

## Обновление README

При изменении шагов пайплайна, стека, расписания или структуры – синхронно обновить `README.md` и, если менялась суть проекта, `PROJECT.md`. Держать актуальными: список шагов, таблицу архитектуры, диаграмму потока, раздел «Запуск локально», список секретов.

## Проверка ошибок

- `run-pipeline.py` не падает на первом сбое: помечает шаг как FAILED и продолжает, итог – ненулевой код возврата и список упавших шагов. Смотреть вывод по секциям `[FETCH] / [FILTER] / [DIGEST]`.
- Пустой дайджест или мало сигналов – проверить, что сбор отработал (`data/raw/` за дату не пуст) и что `OPENAI_API_KEY` задан.
- Ошибки сбора (MTProto) и публикации (Bot API) в облачной песочнице – ожидаемы: эти шаги должны идти на VPS, а не в облаке. В облаке гоняем только фильтр, анализ и композицию.
- Перед коммитом прогнать быстрый прогон с `--skip-fetch` на свежих данных.

## Подготовка к публикации на GitHub

1. Проверить `.gitignore`: `.env`, `settings.local.json`, `.venv/`, кэши – не в git.
2. Убедиться, что в `data/` и логах нет секретов и персональных данных.
3. Обновить `README.md` (скриншоты в `images/`) и `PROJECT.md`.
4. Коммит осмысленным сообщением, затем `git push`.
5. Открыть репозиторий в режиме инкогнито и проверить, что README и скриншоты отображаются.
