## 2. Оценка архитектуры из CLAUDE.md

### Сильные стороны
- Чёткое разделение ответственности: sources / data / digests / agents / scripts
- Правила нейминга устраняют неоднозначность с первого дня
- Дедупликация вынесена в отдельный этап (processed), а не смешана со сбором
- Политика безопасности: токены только через env, raw-данные неизменяемы

### Слабые стороны / пробелы
| Проблема | Риск | Приоритет |
|---|---|---|
| Нет `reports/` в структуре папок | Некуда сохранять аналитику агента | Высокий |
| Нет `.gitignore` и `.env.example` | Случайный коммит секретов | Критичный |
| `settings.local.json` не покрывал `reports/`, `agents/`, `scripts/` | Агент не мог писать скрипты | Высокий |
| Нет схемы данных для `data/processed/` | Разные агенты будут писать в разных форматах | Средний |
| Нет пайплайна запуска (run.py / Makefile) | Нельзя запустить одной командой | Средний |
| Нет `requirements.txt` | Воспроизводимость установки не гарантирована | Средний |

## 4. Рекомендации (следующие шаги)

### Немедленно
1. **Заполнить `.env`** по шаблону `.env.example` — без этого ни один скрипт не запустится
2. **Добавить реальные каналы** в `sources/channels.json` (сейчас 5 заглушек)
3. **Создать `requirements.txt`**:
   ```
   telethon>=1.36
   anthropic>=0.40
   python-dotenv>=1.0
   ```

### Среднесрочно
4. **Пайплайн запуска** — `scripts/run-pipeline.py` или `Makefile`:
   ```
   fetch → filter → digest → alert
   ```
5. **Схема processed-данных** — зафиксировать JSON-схему, чтобы все агенты читали одинаково
6. **Агент оповещений** — `agents/alert-agent.py`: отправка в Telegram при score > 0.9
7. **Хранение состояния** — файл `data/state.json` с last_seen ID по каждому каналу (избежать дублей)

### Долгосрочно
8. **signal_ratio** в `channels.json` — автоматическое обновление после каждого запуска (отношение сигналов к постам)
9. **Архив** — перемещение raw-данных старше 7 дней в архив, чтобы не разрастался репозиторий
10. **Векторный поиск** — ChromaDB/FAISS для поиска похожих событий по истории
