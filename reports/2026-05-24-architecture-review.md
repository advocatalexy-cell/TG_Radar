## 2. Найденные проблемы

### Критичные

**`signal_ratio` всегда `null`**  
В `channels.json` поле `signal_ratio` предусмотрено для автоматического заполнения, но ни `filter-signals.py`, ни другие скрипты его не обновляют. Без этого невозможно автоматически выявлять зашумлённые источники.

**Нет дедупликации постов**  
`fetch-posts.py` каждый раз берёт последние 50 постов без учёта уже обработанных. При двух запусках в день один и тот же пост классифицируется и попадает в дайджест дважды. Нет `data/state.json` с `last_seen` ID по каждому каналу.

### Средние

**Путь к сессии Telegram не привязан к корню проекта**  
```python
# fetch-posts.py:64
TelegramClient("radar_session", API_ID, API_HASH)
```
`"radar_session"` — относительный путь. При запуске из другой директории файл сессии создаётся в CWD, а не в `Radar/`. Сессия теряется.

**Хрупкое извлечение `channel_id` в `filter-signals.py`**  
```python
# filter-signals.py:80
channel_id = raw_file.stem.replace(f"{today}-", "").replace("-raw", "")
```
Работает только если дата в имени файла совпадает с сегодняшней. При обработке вчерашних данных `channel_id` будет содержать дату как часть строки.

**Нет порогового фильтра перед LLM-классификацией**  
Каждый пост отправляется в LLM, включая однострочные объявления и посты без текста (хотя фильтр по `msg.text` в `fetch-posts.py` частично помогает). Посты короче ~100 символов редко несут сигнал — их можно отсеивать без вызова API.

**`stash_link` определяется внутри цикла в `digest-agent.py`**  
```python
# digest-agent.py:133–137
for line in text.splitlines():
    ...
    def stash_link(m):  # переопределяется на каждой итерации
```
Функция переопределяется на каждой итерации цикла. Работает, но создаёт замыкание на переменную `links` из внешнего цикла — это неочевидное поведение.

### Низкие

**Дайджест берёт топ-100 сигналов без порогового фильтра по score**  
Сигналы с `score: 0.1` попадают в дайджест наравне с `score: 0.9`. Рекомендуется минимальный порог (например, `score >= 0.5`).

**Архитектурный документ (CLAUDE.md) не отражает реальные модели**  
Таблица моделей в предыдущем отчёте указывала Claude Haiku для классификации. Фактически используется Groq. CLAUDE.md это не описывает.

## 3. Выполнено в этом обзоре

### Создан `scripts/run-pipeline.py`

Оркестратор, запускающий все три шага последовательно:
```
python scripts/run-pipeline.py
```
Флаги для пропуска отдельных шагов:
```
python scripts/run-pipeline.py --skip-fetch --skip-filter
```
При падении любого шага выводит предупреждение и продолжает (не блокирует). Финальный exit-код ненулевой если хотя бы один шаг упал.

### Создан `requirements.txt`

```
telethon>=1.36
groq>=0.9
openai>=1.30
python-dotenv>=1.0
```

## 4. Приоритизированные улучшения

### Немедленно (блокируют качество данных)

1. **Добавить `data/state.json`** — хранить `last_seen` message ID по каждому каналу. В `fetch-posts.py` передавать `min_id=last_seen` в `iter_messages`. Устраняет дубли.

2. **Автообновление `signal_ratio`** — в конце `filter-signals.py` вычислять `signals/total` по каждому каналу и записывать в `channels.json`. Позволит ревьювить зашумлённые источники.

3. **Зафиксировать путь к сессии**:
   ```python
   SESSION = str(ROOT / "radar_session")
   TelegramClient(SESSION, API_ID, API_HASH)
   ```

### Среднесрочно

4. **Порог score в дайджесте** — фильтровать `signals` по `score >= 0.5` перед отправкой в LLM.

5. **Пре-фильтр коротких постов** — пропускать классификацию для постов короче 80 символов, автоматически помечать как `signal: false`.

6. **Исправить `channel_id` в `filter-signals.py`** — использовать regex вместо `replace`:
   ```python
   import re
   channel_id = re.sub(r"^\d{4}-\d{2}-\d{2}-(.+)-raw$", r"\1", raw_file.stem)
   ```

### Долгосрочно

7. **Архив raw-данных** — перемещать файлы старше 7 дней в `data/archive/`.
8. **Векторный поиск** — ChromaDB для поиска похожих сигналов по истории (тренды во времени).

## 5. Схема пайплайна (актуальная)

```
sources/channels.json
        │
        ▼
scripts/fetch-posts.py  ──────────► data/raw/YYYY-MM-DD-{channel}-raw.json
   (Telethon, user session)
        │
        ▼
scripts/filter-signals.py  ────────► data/processed/YYYY-MM-DD-{channel}-processed.json
   (Groq llama-3.3-70b → fallback OpenAI)
        │
        ▼
agents/digest-agent.py  ───────────► digests/YYYY-MM-DD-digest.md
   (Groq llama-3.3-70b → fallback OpenAI)       │
                                                 ▼
                                        Telegram Bot (HTML, чанки по 4096)
```

**Оркестратор:** `scripts/run-pipeline.py` (создан в этом обзоре)
